Ovation Web Frontend Architecture

Next.js 16 App Router · React 19 · React Query · Zustand · better-auth · Tailwind 4

Overview

Ovation Web is a Next.js 16 App Router application in a pnpm monorepo alongside ovation-api. It serves three distinct audiences: couples managing their wedding event dashboard, guests submitting messages and media via a public link, and pro users (planners) managing multiple events.

Next.js App Router

Nested layouts, server components, locale-segmented routing via next-intl

Feature-Based Structure

Self-contained feature folders under src/features/ with pages, hooks, and components co-located

Dual State Strategy

React Query for server state · Zustand for multi-step form flows

Workspace UI Packages

@ovation/ui, @ovation/icons, @ovation/illustrations shared across apps

Server vs Client Rendering

The app follows Next.js App Router conventions: server by default, client by exception.

PatternWhereWhy
Async Server ComponentLayout files, page shellsAuth checks, initial data fetch, no JS bundle cost
"use client" componentInteractive regions, forms, hooksEvent handlers, React state, browser APIs (MediaRecorder)
React Query prefetchServer components passing dehydrated stateAvoid waterfall — data arrives with the HTML
Streaming / SuspenseHeavy list sectionsPage shell renders immediately, data streams in
The split is intentional: layouts fetch auth + event data on the server (zero client JS, fast first paint), while interactive features like message playback or media recording are client components that hydrate after.

Middleware

src/middleware.ts runs on every request before routing. It handles locale detection and negotiation via next-intl, and injects the [locale] segment into all matched routes. Unauthenticated redirects are handled inside layout server components, not middleware.

Auth & Sessions

Authentication is powered by better-auth with a PostgreSQL backend. There are two separate instances: a server instance (Node.js only) and a browser client.

Server Instance src/lib/auth/better-auth.ts

  • PostgreSQL connection pooling via DATABASE_URL
  • Email/password auth with email verification (links via Resend)
  • OAuth: Google, Apple with account linking
  • Max 5 concurrent sessions per user
  • Session lifecycle hooks → auth event logging to DB
  • CSRF token generated per session, rotated on sign-in
  • Custom Zod validators: password 8–128 chars
  • Risk scoring via HMAC-SHA256 of IP + User-Agent (Pepper from env)

Browser Client src/lib/auth/client.ts

import { createAuthClient } from 'better-auth/react';

export const authClient = createAuthClient({ baseURL: process.env.NEXT_PUBLIC_API_URL });
// Exports: signIn, signUp, signOut, useSession, getSession

API Layer

All communication with ovation-api goes through typed client modules in src/lib/api/. There is no direct fetch() in components or hooks.

Base Fetch Wrapper src/lib/api/client.ts

  • clientFetch<T>(path, init) — browser-side fetch with credentials + CSRF injection
  • clientFetchPaginated<T>(path) — returns { items: T[], nextCursor }
  • Auto-prefixes path with /api/v1
  • On 403 with CSRF error code → fetches fresh CSRF token and retries once automatically
  • Throws typed ApiError (status, code, message, details) on non-2xx

Server Fetch src/lib/api/server.ts

  • apiFetch<T>(path, init) — Node.js fetch for server components
  • Copies the incoming request's session cookies forward
  • Sets Accept-Language header from getLocale()

Public Client src/lib/api/public-client.ts

Used by the guest submission pages. No auth, no CSRF — just a plain fetch to the public API endpoints: getEvent(), uploadUrls(), createMessage(), getGallery(), recordInvitationOpen().

React Query

React Query is the primary source of truth for all backend data in client components. The query client is configured in src/lib/query/QueryProvider.tsx and mounted in the locale layout via AppProviders.

Client Config

  • Stale time: 30 seconds
  • Refetch on window focus: disabled
  • Retry: 2 retries for 5xx only (4xx errors are not retried)

Query Key Factory src/lib/query/keys.ts

export const queryKeys = {
  messages: (eventId: string) => ['messages', eventId],
  message:  (id: string)      => ['messages', id],
  gallery:  (eventId: string) => ['gallery', eventId],
  orders:   ()                  => ['orders'],
  jobs:     (jobId: string)    => ['jobs', jobId],
};

Form Handling

All forms use react-hook-form with Zod schemas via @hookform/resolvers/zod (standardSchemaResolver). No class-validator or class-transformer.

1️⃣
Define Zod schema
*.schema.ts in the feature folder. Schemas accept a t translation function for i18n error messages: getSignUpSchema(t).
2️⃣
useForm hook
useForm({ resolver: standardSchemaResolver(schema), mode: 'onTouched', reValidateMode: 'onChange' }). Validation on blur, re-validate on change after first touch.
3️⃣
Register inputs
Simple inputs: <Input {...register('field')} />. Complex fields (Checkbox, Select): <Controller> wrapping the UI component.
4️⃣
Submit handler
Calls API client method. On success: update Zustand store if cross-step data, then navigate. On API error: map ApiError.code to i18n string and call setError().
5️⃣
Display errors
errors.field?.message rendered inline below each field. Server-side errors mapped to field errors or a root-level form error.

Key User Flows

Sign-Up Flow

1
/sign-up — Email + password form. On submit: authClient.signUp.email(). Email stored in useSignUpStore.
2
/verify — OTP input. better-auth sends verification email via Resend. On success: auto sign-in + navigate.
3
/role — Select Couple or Pro. Sets accountType in store + calls user update API.
4
/plans — Select subscription tier. On select: store selectedPlan, navigate to checkout or directly to create.
5
/create — Event creation wizard. Store reset on complete. Navigate to /home.

Create Event Flow

All wizard steps share data via useCreateEventStore. The useSlugChecker hook debounces a uniqueness check against the API as the user types.

Guest Submission Flow

The guest flow lives under /g/[slug] (a public route — no auth). It uses the public-client.ts fetch wrapper, no CSRF. The Turnstile CAPTCHA widget is mounted to satisfy the PublicSubmissionGuard on the API side.

1
Landing — Load event by slug (server). Render GuestHero + PolaroidPair branding. Record invitation open.
2
Compose — Guest types name + message. Optionally records audio/video via useAudioRecorder / useVideoRecorder (MediaRecorder API). Optionally attaches photos.
3
Media upload — For each file: call uploadUrls() to get pre-signed S3 URL. PUT file directly to S3. Collect returned media IDs.
4
Review — Preview name, message, and media. Turnstile CAPTCHA solved in background.
5
Submit — POST /public/messages with Turnstile token + dwell-time header + media IDs. On success: confirmation screen.

Media Uploads

The frontend never sends binary files to the API server. It uses a pre-signed URL pattern:

  1. Request a pre-signed S3 PUT URL from the API (POST /storage/upload-url).
  2. PUT the file bytes directly to S3 from the browser — API server is not in the data path.
  3. Call the API confirm endpoint with the returned S3 key.
  4. API enqueues thumbnail generation and HEIC conversion jobs in BullMQ.

The usePhotoUpload hook handles progress tracking, retry on network failure, and cancellation. useVideoRecorder and useAudioRecorder wrap the browser MediaRecorder API with chunked recording and local preview.

Internationalisation

Locale routing is handled by next-intl. Supported locales: en, fr, nl, de, es, it. The locale is the first segment in every URL: /en/home, /fr/home.

FilePurpose
src/i18n/routing.tsLocale list + default locale config for next-intl
src/i18n/navigation.tsExports locale-aware Link, useRouter, redirect, usePathname
src/lib/routes.tsTyped route helper — always call via i18n navigation exports, never raw next/navigation
[locale]/layout.tsxLoads translation JSON for the locale, mounts NextIntlClientProvider

Zod schemas accept a t function so validation error messages are localised. API error codes are mapped to translation keys client-side before display.
